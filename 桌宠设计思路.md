# 桌宠设计思路文档

## 1. 项目定位

**项目名称**：DeskPet

这是一个面向 Windows 的轻量桌面宠物应用。当前目标不是一开始追求 AI 对话、云端社区或复杂游戏系统，而是先把“桌宠真的能在桌面上显示、切换、互动、保存设置”这条核心体验做稳定。

当前技术路线已经确定为：

```text
Tauri 2 + TypeScript + Vite + PixiJS
```

核心原则：

- 本地优先，无独立后端。
- 宠物以本地宠物包形式存在。
- 动画统一走 GIF 拆帧后合成 spritesheet 的流程。
- 先保证单宠物窗口稳定，再做随机行为、状态机、托盘和导入。
- 桌宠资源可以本地自用；涉及第三方 IP 的素材不建议随项目公开分发或商用。

## 2. 当前项目实际状态

截至当前版本，项目已经完成了一个可运行的桌宠 MVP。

### 已完成能力

- PixiJS 渲染宠物 spritesheet 动画。
- 本地宠物包加载：`public/pets/<pet-id>/pet.json`。
- 宠物台词加载：`lines.json`。
- 宠物选择下拉菜单。
- 动作按钮：跳跃、睡觉、开心、专属。
- 右键/控制菜单雏形。
- 用户缩放选择。
- 退出按钮。
- 当前宠物、缩放、窗口位置保存。
- GIF 动作素材拆帧合成 spritesheet 的制作流程。

### 当前已启用宠物

当前 `public/pets/pets.json` 中启用：

```json
[
  "moyu-cat",
  "offwork-hero",
  "goldpotato"
]
```

其中：

| 宠物 ID | 显示名 | 状态 | 资源来源/说明 |
|---|---|---|---|
| `moyu-cat` | 摸鱼猫 | 示例资源包 | 原创示例宠物 |
| `offwork-hero` | 下班超人 | 示例资源包 | 原创示例宠物 |
| `goldpotato` | 金土豆 | 旧资源包保留 | 后续可迁移到 GIF 流程 |

### 当前标准宠物规格

后续新增宠物建议统一使用：

```text
frameWidth: 192
frameHeight: 208
spritesheet: 1536 x 1664
layout: 8 列 x 8 行
scale: 0.95
```

每一行动作约定：

| Row | 动作 | 对应 AnimationId |
|---:|---|---|
| 0 | 待机 | `idle` |
| 1 | 向左跑 | `walkLeft` |
| 2 | 向右跑 | `walkRight` |
| 3 | 挥手/点击反馈 | `touch` / `happy` |
| 4 | 跳跃 | `jump` |
| 5 | 等待/拖拽/低功耗 | `drag` / `sleep` |
| 6 | 失败/落地 | `drop` |
| 7 | 专属动作 | `special` |

## 3. 当前交互状态与已知问题

### 当前交互策略

为了先恢复稳定性，当前交互暂时采用较宽松方案：

- 左键点击桌宠窗口区域：触发 `touch`。
- 左键拖动桌宠窗口区域：触发窗口拖拽。
- 右键桌宠窗口区域：打开控制菜单。
- 控制菜单中的动作按钮可以正常触发动画。
- 控制菜单不再依赖 hover 自动弹出，避免遮挡宠物点击。

### 已经尝试过但暂缓的方案

曾尝试做“只点击宠物可见像素才响应”：

- 使用 Pixi `renderer.extract.pixels()` 读取当前渲染像素。
- 使用 spritesheet 原图 alpha 读取命中像素。
- 使用宠物中心矩形 hit test。
- 给透明窗口增加极低透明背景以改善 WebView2 命中。

这些方案在 Tauri + WebView2 + 透明窗口组合下表现不稳定，出现过：

- 左右键点击宠物无反应。
- 无法拖动宠物。
- 半透明背景方块可见。
- 事件没有稳定进入 canvas/WebView。

因此当前先回退到稳定交互，后续需要重新设计命中系统。

### 当前已知问题

1. **透明区域点击范围仍偏大**  
   现在恢复了可用交互，但还没有实现“只点宠物身体才响应”。

2. **透明窗口无法真正点击穿透**  
   Tauri 窗口仍是 360x360 透明矩形。即使 JS 不触发动作，透明区域也可能挡住桌面点击。

3. **右键菜单仍是 HTML dock，不是系统原生菜单**  
   当前菜单能用，但还不是完整托盘/系统菜单。

4. **状态机还没有正式引入**  
   当前仍是事件直接播放动画，动作优先级、打断规则、sleep 唤醒等还不完善。

## 4. 技术架构

### 前端层

主要文件：

```text
src/main.ts
src/pet-player.ts
src/pet-loader.ts
src/types.ts
src/styles.css
index.html
```

职责：

- `main.ts`
  - 初始化 Pixi Application。
  - 加载宠物列表。
  - 切换宠物。
  - 绑定点击、拖拽、右键、按钮事件。
  - 保存/恢复 localStorage 设置。
  - 调用 Tauri window API。

- `pet-player.ts`
  - 加载 spritesheet。
  - 根据 `pet.json` 切帧。
  - 使用 Pixi `AnimatedSprite` 播放动画。
  - 管理宠物基础缩放和用户缩放。
  - 切换宠物时释放旧资源。

- `pet-loader.ts`
  - 加载 `pets.json`。
  - 加载单个宠物的 `pet.json` 与 `lines.json`。
  - 解析宠物资源路径。

- `types.ts`
  - 定义 `AnimationId`、`PetManifest`、`PetAnimation` 等类型。

### Tauri 层

主要文件：

```text
src-tauri/tauri.conf.json
src-tauri/capabilities/default.json
src-tauri/src/main.rs
```

当前能力：

- 透明窗口。
- 无边框。
- 置顶。
- 跳过任务栏。
- 原生窗口拖拽。
- 关闭窗口。
- 获取/设置窗口位置。

当前 Rust 主入口仍保持最小化，没有自定义 command。

    "walkRight": { "row": 2, "frames": 8, "fps": 6, "loop": true },

## 6. GIF 宠物制作流程

后续统一推荐使用 GIF 动作素材，不再优先使用单张静态图。

### 输入素材要求

每只宠物建议准备 8 个 GIF：

```text
idle
running-left
running-right
waving
jumping
waiting
failed
review
```

要求：

- 所有 GIF 尺寸一致，推荐 `192 x 208`。
- 背景透明。
- 每个动作 4-8 帧。
- idle / run / touch / jump 至少要有。
- 文件名尽量规范，方便脚本批处理。

### 输出流程

1. 检查 GIF 尺寸、帧数、透明度。
2. 使用 PowerShell + System.Drawing 拆帧。
3. 合成 `spritesheet.png`。
4. 取 idle 第一帧生成 `preview.png`。
5. 写 `pet.json`。
6. 写 `lines.json`。
7. 把宠物 id 加入 `public/pets/pets.json`。
8. 运行 `npm run build` 验证。
9. 运行 `npm run desktop` 实机测试。

### 当前合成规则

```text
列数：8
行数：8
单帧：192 x 208
图集：1536 x 1664
格式：PNG 透明图
```

PNG 比 WebP 文件可能稍大，但处理简单、透明稳定。后续如果包体需要优化，再统一压缩为 WebP。

## 7. 动作与状态机设计

### 当前已支持动作

```ts
idle
walkLeft
walkRight
touch
drag
jump
drop
sleep
happy
special
```

### 当前流转方式

当前还不是完整状态机，而是事件直接调用：

```text
点击 -> touch -> idle
拖动 -> drag -> drop -> idle
按钮 sleep -> sleep
按钮 happy -> happy -> idle
按钮 special -> special -> idle
```

### 下一步状态机目标

后续应增加轻量状态机：

```text
idle
acting
dragging
sleeping
```

需要定义：

- 哪些动作可以打断当前动作。
- sleep 是否只能点击唤醒。
- drag/drop 优先级最高。
- special 播放时是否允许再次触发。
- 随机 idle 行为是否能被用户点击打断。

## 8. 下一阶段开发优先级

### P0：先稳定桌宠交互

1. **保持当前可用交互稳定**
   - 点击、拖动、右键菜单不能再被 hit-test 改坏。
   - 修改交互前先做最小可复现测试。

2. **重新设计命中区域**
   - 不再依赖 WebGL 实时像素读取。
   - 推荐在 `pet.json` 中增加简单 `hitbox`：

   ```json
   "hitbox": {
     "x": 32,
     "y": 16,
     "width": 128,
     "height": 176
   }
   ```

   - hitbox 以单帧 192x208 坐标为基准。
   - 所有宠物先用默认 hitbox，再逐个微调。
   - 先实现“宠物附近可点，远处不可点”，不要追求像素级精度。

3. **处理透明窗口点击穿透问题**
   - 当前 Tauri 窗口仍是 360x360 透明矩形。
   - 后续研究 Tauri/WebView2 是否能动态设置 click-through。
   - 目标是非宠物区域尽量不挡桌面操作。

### P1：让宠物更像“活的”

1. **随机待机行为**
   - 每隔 20-60 秒随机触发 idle 台词、happy、special 或短动作。
   - 必须有冷却，避免打扰。

2. **连续点击互动**
   - 连点 3 次触发特殊台词。
   - 连点 5 次触发 happy/special。

3. **轻量状态机**
   - 引入 `idle / acting / dragging / sleeping`。
   - 防止动作互相打断导致状态混乱。

4. **宠物专属台词风格**
   - 摸鱼猫：轻松、打工、陪伴。
   - 下班超人：元气、幽默、鼓励。
   - 光翼仙人：温和、治愈、提醒。

### P2：工程化与资源流程

1. **新增宠物生成脚本**
   - 把当前临时 PowerShell 拆帧逻辑整理成项目脚本：

   ```text
   scripts/generate-gif-pet.ps1
   ```

   - 输入源目录和 pet id，自动生成 spritesheet、preview、pet.json 模板。

2. **宠物包校验脚本**
   - 检查 `pet.json` 字段是否完整。
   - 检查 spritesheet 是否存在。
   - 检查 frameWidth/frameHeight 是否匹配。
   - 检查 row/frames 是否越界。

3. **统一资源清理**
   - `public/pets` 中只放运行时需要的正式资源。
   - 原始 GIF/PNG 可放 `pets/` 或 `assets-source/`。
   - 避免中间产物进入正式包。

### P3：桌面能力增强

1. 系统托盘菜单。
2. 开机自启。
3. 设置窗口。
4. 宠物选择面板，使用 preview 图展示。
5. 性能模式：省电 / 标准 / 流畅。
6. 打包发布流程优化。

## 9. 性能与内存策略

桌宠是常驻应用，需要持续关注资源占用。

当前资源特点：

- GIF 标准宠物 spritesheet 为 `1536 x 1664`。
- 解码后单图约：

```text
1536 x 1664 x 4 ≈ 10 MB
```

当前策略：

- 只加载当前宠物完整 spritesheet。
- 切换宠物时销毁当前 Pixi 对象。
- 调用 `Assets.unload` 释放旧资源。
- 不一次性加载所有宠物图集。
- preview 后续应使用独立小图。

后续优化方向：

- idle/sleep 状态降低动画频率。
- 无动作时考虑降低 ticker 频率。
- 避免过大 spritesheet。
- 打包后测试正式版内存，不只看 `npm run desktop` 开发模式。

## 11. 推荐开发流程

后续每个功能都按这个流程走：

1. 明确目标和影响范围。
2. 先保证现有交互不回归。
3. 小步修改，不一次改太多交互核心。
4. 每次修改后运行：

```bash
npm run build
```

5. 涉及桌面行为时运行：

```bash
npm run desktop
```

6. 实机验证：
   - 点击。
   - 拖动。
   - 右键菜单。
   - 切换宠物。
   - 缩放。
   - 退出和重启恢复。

7. 如果是资源新增：
   - 检查 GIF 尺寸和帧数。
   - 生成 spritesheet/preview。
   - 加入 `pets.json`。
   - 验证切换和动作。
